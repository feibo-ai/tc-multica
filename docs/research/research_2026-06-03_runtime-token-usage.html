RESEARCH · 研究

个人 × runtime 全量 token 用量统计

Phase RPI Research日期 2026-06-03项目 Multica 迭代DRI 曾振华

问题

Question
当前 Multica 的 token usage 统计逻辑是怎样的（数据来源、归集路径、task_usage_hourly / runtime usage 的覆盖范围）？如何新增「以 个人 × runtime 为粒度、覆盖本机所有对话（含未挂载到 Multica 的 agent）」的统计维度？重点验证用户假设：能否通过 Multica CLI / daemon 解析本地对话记录（如 Claude Code transcript JSONL）采集全部 token 用量，打通 采集 → 归属(person × runtime) → 存储 → 展示，并处理与既有统计的去重 + 隐私边界。
能力边界判定（SOP 强制第一问）： 在 AI 能力边界内 ✓。本任务是「读懂现有代码 + 设计数据采集扩展」，纯属强项，不触边界（SOP ❌6 无风险）。

发现 · 一

Findings — 当前统计逻辑（现状全图）

用量统计 100% 与「任务」耦合：只有经 agent_task_queue 派发、由 daemon 执行的任务才会产生用量记录。整条链路只有一条，无旁路。

写入路径（唯一）

  • agent CLI（claude/codex…）在 daemon 下运行 → 累积 result.Usage（pkg/agent/claude.go:246-252,391-401；结构 pkg/agent/agent.go:84-99）
  • 任务终态（成功/失败/取消都报）→ d.client.ReportTaskUsage(task.ID, result.Usage)（internal/daemon/daemon.go:2067-2071）
  • POST /api/daemon/tasks/{taskId}/usage（router.go:308）→ handler ReportTaskUsage（internal/handler/daemon.go:1752-1783，经 requireDaemonTaskAccess 校验 workspace 归属）
  • UpsertTaskUsage 写入 task_usage（db/queries/task_usage.sql:1-14）。载荷只有 provider/model/*_tokens，无 runtime_id/agent_id/user_id/session_id —— 其它维度全靠后续 JOIN 任务行还原。

表结构

  • task_usage（migration 032）：UNIQUE(task_id, provider, model)；列 = task_id(FK→agent_task_queue)、provider、model、input/output/cache_read/cache_write_tokens。无 runtime/agent/user/workspace 列。
  • task_usage_hourly（migration 101）：rollup 表，所有用量报表的唯一真相源。键 = hour(UTC) × workspace × runtime × agent × project × provider × model。无 user / person 维度。配套 task_usage_hourly_dirty（失效队列）+ rollup_state（水位）。

归集（rollup）

  • SQL 函数 rollup_task_usage_hourly()（migration 102）：advisory-lock、读水位、整桶重算（非增量），JOIN task_usage → agent_task_queue → agent → issue 求和，过滤 atq.runtime_id IS NOT NULL。
  • 近期 commit「run task_usage_hourly rollup in-process」= cmd/server/usage_rollup.go:36-92，作为 goroutine 每 60s tick 一次。原因：Zeabur 的 pgvector Postgres 没有 pg_cron，必须进程内跑（这条约束对新方案同样成立）。

读取路径

  • Dashboard（/api/dashboard/*）：按 date×model、agent×model；Runtime 详情（/api/runtimes/{id}/usage,by-agent,by-hour）；Issue 用量。
  • 现有可聚合维度：workspace / runtime / agent / project / model / provider / hour / date。无 per-person 聚合。
覆盖边界（最关键）：用量只对走 agent_task_queue 的工作记账。task_usage.task_id 是 NOT NULL FK，无任务行 ⇒ 无用量行。代码已自陈这个盲区 —— handler/runtime.go:84-87：用量「scoped to Daemon-executed tasks only (i.e. excludes users' local CLI usage of the same tool)」。未挂载到 Multica 的本地对话，今天产生 0 行，对所有报表不可见。

发现 · 二

Findings — runtime 身份 & person 归属
重大利好：「person × runtime」的数据模型已经存在。 agent_runtime.owner_id UUID REFERENCES "user"(id)（migration 032_runtime_owner.up.sql:1，回填为 workspace owner）。即 个人 = runtime.owner_id，runtime = agent_runtime.id。已挂载的 runtime 不需要新建身份表。
  • runtime = agent_runtime 行（migration 004）。runtime_mode ∈ {local, cloud}（本地 daemon vs 云端）；provider（claude/codex/multica_agent/…）；daemon_id（哪台机器）；owner_id、status、last_seen_at、visibility。唯一键 (workspace_id, daemon_id, provider)。
  • person = user，经 member(workspace_id, user_id, role) 入 workspace。
  • daemon 今天只：注册/心跳 runtime、轮询队列、执行被派发的任务。不观察用户自己起的 ad-hoc 本地会话。daemon disk-usage 只扫 ~/multica_workspaces，从不碰 ~/.claude。

用户假设「解析本地对话记录」可行：原始数据齐全，且仓库已有两套解析器（只是被绑死在任务执行链路里）。

发现 · 四

Findings — 采集→server 链路 & 自然扩展点

数据流（现状）：

agent CLI → result.Usage → daemon.ReportTaskUsage(task.ID) → POST /api/daemon/tasks/{taskId}/usage → UpsertTaskUsage → task_usage → [每60s] rollup_task_usage_hourly() → task_usage_hourly → 读端
  • 唯一 ingestion = POST /api/daemon/tasks/{taskId}/usage，URL + DB FK 双重 task 绑定。没有通用「上报用量事件」端点。（internal/metrics 是 Prometheus 运维遥测、internal/analytics PostHog 无 token 数 —— 都不是用量入口。）
  • daemon 鉴权：Bearer mdt_/mul_/JWT（middleware/daemon_auth.go）。注册 POST /api/daemon/register upsert agent_runtime 并落 owner_id。心跳不带用量。

自然扩展点（仅描述，不设计）

  1. 新建 runtime 级写端点（而非 task 级），如 POST /api/daemon/runtimes/{runtimeId}/usage，复用 requireDaemonRuntimeAccess（daemon.go:73）—— 免费拿到 workspace + owner(person) 归属。
  2. 新建一张 task-less 用量表，键 (runtime_id [+ session/conversation/request id], provider, model, bucket)，不强制 task_id/agent_id —— 因为现有 task_usage(_hourly) 的 NOT NULL 约束装不下无任务的行。
  3. daemon 侧采集点：仿 diskusage.go 扫描任务目录的方式，做一个 runtime 级的本地会话清扫（增量按 mtime/offset），就地解析、就地去重、只上报数字。
  4. 读/rollup 复用：进程内分钟级 rollup worker、tz 机器、CLI 表格渲染均可原样复用。

发现 · 五

Findings — 坑 / 约束 / 前人方案

三个必须正面处理的坑

  1. 双重计数（mounted run vs 解析 transcript） —— 高危但可解。现有 task_usage 键 (task_id, provider, model)，不带 session_id/message_id/request_id，无共享键可去重。好消息：daemon 已存了每个任务的 Claude session_id（agent_task_queue.session_id），而 transcript 文件名就是该 session_id → 去重 = 「跳过 session_id 已被任务计过的 transcript」。难点：一个 session 跨多任务续用（prior_session_id）、文件内一个请求产生 2-10 行同 requestId。结论：需要在用量写入链路加 (session_id, message_id, request_id) 去重键。
  2. 100× 漏算陷阱 —— 高危、极易写错。约 75% 的 Claude JSONL assistant 行 input_tokens 是 0/1 的流式占位符，从不回填；朴素求和会把 input 少算 100-174×、output 少算 10-17×。正解：每个 requestId 取最终/最大用量对象，而非逐行求和（ccusage 法）；更省事的替代（agent-usage 法）：直接 cache_creation_input_tokens == nil ⇒ 跳过 滤掉流式块 —— 只有终态消息带该字段，无需追 requestId。两法皆可、Plan 选一；都依赖「终态消息一定带 cache_creation」这个假设。cache 字段从头可靠。
  3. 隐私 —— transcript 含完整 prompt/代码/工具 I/O。本功能只需 session_id + timestamp + model + provider + 四个 token 数。绝不读取/上传 message.content。就地解析、只上报数字聚合。仓库目前无本地日志的隐私条款 —— Plan 需新立这条边界。

其它风险

  • schema drift：JSONL 是未公开、版本易变的内部格式（Anthropic 有 open issue #33978 要做一等 claude usage）。解析必须逐行 fail-soft、缺字段当 0（对齐仓库「Enum drift downgrades, not crashes」doctrine）。
  • 归属歧义：owner_id 只覆盖已挂载 runtime；未挂载/共享机器需定 (machine→person) 绑定。本机 git config user.email 为空，该归属路径不可靠。
  • 覆盖面：Claude ≠ 全部工具，Codex 等格式各异 —— 建议 v1 先 Claude，其余做可插拔 adapter。

前人方案

  • briqt/agent-usage（github，Go · 用户点名）：单机单用户版 ccusage ×6 工具（Claude/Codex/OpenClaw/OpenCode/Kiro/Pi）。一个 Go 二进制扫本机日志 → SQLite → 内嵌 dashboard，GET-only、无鉴权、无 ingestion、无跨机同步、schema 零 user/person/host 维度。采集层机制可直接借鉴，但「归属」它完全没做 —— 归属正是 Multica 的独有难点 + 价值。可借鉴 5 件：① 漏算用结构过滤替代 requestId 去重（cache_creation_input_tokens == nil ⇒ skip + model=="<synthetic>" ⇒ skip）；② 增量 tailing file_state(path,last_offset,scan_context) seek 续读、缓存解析态；③ 幂等去重 INSERT OR IGNORE + 复合唯一索引 (session_id,model,timestamp,in,out)（含 timestamp+token，罕见同值会误撞掉一条）；④ 成本层解耦：只存裸 token，按 litellm 价表每小时同步并回填历史；⑤ Collector 接口 Scan() error 可插拔（Codex 累计取差 input−cached、OpenCode 读 SQLite 源、Kiro 只能按 % 估算）。
  • ccusage（github）：读 ~/.claude/projects/**/*.jsonl，按 messageId:requestId 哈希去重（一个请求 2-10 行）。单机参考实现。
  • Claude Code 原生 OTEL（docs）：每请求发 claude_code.token.usage，带类型 + session.id + user.email + query_source。是 log-parse 的真正替代路线：来自终态响应（无 100× 漏算）、不含 transcript 内容（隐私干净）、自带 person 归属；代价是每用户要配 OTEL_EXPORTER_OTLP_ENDPOINT，且只覆盖会发 OTEL 的工具。
  • cache 计费坑（litellm #19680）：cache-read 按折扣价计费，必须与普通 input 分开算（仓库 schema 已分 cache_read/cache_write ✓）。

仓库硬约束（Plan 必须遵守）

  • 多租户：所有 query 过滤 workspace_id，X-Workspace-ID 路由（CLAUDE.md:422）。新表/新 query 必须带并过滤 workspace_id。
  • Handler UUID 解析约定（#1661）：写库前先经 loader 校验，再用 entity.ID；新 ingest 端点照此顺序。
  • API Response Compatibility（#2143/2147/2192）：任何 UI 消费的新端点须 parseWithFallback + zod schema + 一个 malformed-response 测试，同 PR 交付。
  • 包边界：transcript 解析是 Go（CLI/daemon）的事，绝不进 packages/core；UI 进 packages/views、web+desktop 双端接线。
  • 复用既有管线、不分叉：task_usage_hourly 是唯一真相源（其 PK 已含 runtime/agent/project/provider/model）。新用量应汇入该 rollup 体系，而非另起平行表。（与「新表装 task-less 行」需在 Plan 里调和 —— 见开放问题。）

待解问题

Open Questions — Plan 前必须定
  1. 采集源之争：解析 JSONL vs 接 OTEL？ OTEL 绕开 100× 漏算、零内容外泄、自带 user.email；但要用户配 exporter。JSONL 被动可用但脆且漏算。可能 v1 JSONL + 后续 OTEL。
  2. 未挂载会话的 person/runtime 归属：transcript 无身份。按 (machine daemon_id → runtime owner_id)？按本地 git email（本机为空，不可靠）？还是 daemon 盖上登录的 Multica user？多人共用一机今天无解。
  3. 未挂载会话算哪个 runtime？ 用户交互式 Claude Desktop 可能根本没有 agent_runtime 行。是否给每个 (machine × provider × user) 造一个合成/隐式 runtime，还是归到 catch-all？
  4. 存储形态：扩 task_usage（需 nullable task_id，破坏 FK/unique）？还是新 session_usage/runtime_usage 表 + 汇入 task_usage_hourly？rollup PK 是否加 person 维度，还是经 runtime.owner_id 派生？
  5. 去重权威：daemon 上传前剔除已计 session，还是 server 端 rollup 时按 session_id 对账？两种都要求把 session_id 加进用量写入链路。
  6. 回填窗口：本机 ~/.claude/projects 存了数月历史 —— 一次性 backfill 还是只向前？（注意：Codex total_token_usage 累计「末值取胜」，Claude 用量逐消息「需求和」，两源聚合数学不同。）
  7. 多工具范围：Claude-only v1，还是 Codex/其它从一开始就做 pluggable adapter？

推荐方向

Recommended Approaches — 选项，非决策（决策属 Plan session）

方向 A —— daemon 本地扫描器（用户假设的直球落地）

daemon 定期清扫 ~/.claude/projects(+~/.codex/sessions)，就地解析用量（每 requestId 取终值）、按 session_id 对已计任务去重、归到 runtime.owner_id，经新 runtime 级端点上报到新 task-less 用量表，再汇入 task_usage_hourly rollup。
优：被动零配置、复用 owner_id 人链接、复用 rollup 模式、仓库已有解析器可改。劣：要正面扛 100× 漏算 + schema drift + 去重 + 隐私（碰内容文件）。
现成参考：briqt/agent-usage 就是这套采集层的 Go 实现（扫描 + 增量 tailing + 幂等去重 + 6 工具 + 成本回填），可直接对照借鉴 —— Multica 只需补它缺的那半：transport + 鉴权 + person×runtime 归属，而这半 Multica 的 daemon（已有 owner_id + Bearer 鉴权 + rollup）恰好已具备。A 的实现风险因此显著下降。

方向 B —— 接 Claude Code 原生 OTEL

引导用户把 Claude Code 的 OTEL exporter 指向 Multica 接收端，吃 claude_code.token.usage 指标（终态、带类型、含 session.id + user.email）。
优：无漏算、零内容外泄（隐私干净）、自带 person 归属、字段稳定。劣：每用户要配 OTEL_EXPORTER_OTLP_ENDPOINT（配置摩擦）；只覆盖发 OTEL 的工具（Claude Code 有，Codex 待查）；需建 OTLP 接收端点。

方向 C —— 分阶段混合

v1 = 方向 A 但只做 Claude JSONL、按 session_id 去重、归属用 daemon owner；v2 加 Codex adapter；可选 v3 引入方向 B 的 OTEL 作为高保真路径/校验。
优：最快见效、风险分摊、appetite 可控。劣：先吃 JSONL 的全部坑。

给 Plan 的种子判断（不是决策）：「person × runtime」骨架已有（runtime.owner_id），最大未知不在「存哪」而在「未挂载会话如何归到一个 person/runtime」+「去重与漏算的正确性」。建议 Plan 先在 A vs B 之间拍源，再定存储与归属。
