目标

Goal
tc-multica 前端三块改造，一个 plan 分 3 波次：① 新增「团队」顶部 tab 总览页——全员默认落地、全员可见全员，每个成员一张高密度信息卡（封闭清单维度，全通俗中文命名 + 每指标 hover 解释，无英文行话），让 boss 一眼看清每个人在干什么、当前状态如何；② 侧边栏「搜索 / 新建issue」从顶部移到底部 + 新增可持久化的收缩功能；③ 项目页主从布局——点 issue 后保留「侧边栏 + 项目栏 + issue详情」三栏，不再整页跳走丢失列表栏。

完成标准

  • 波次1·后端：GET /api/team/overview 端点活、挂 RequireWorkspaceMember 受保护组、所有 query 过 workspace_id，一次返回全员卡数据（成员/角色/小队 · 项目 lead-DRI-参与数 · issue 按状态分布 · agents+运行中数 · autopilot数 · 复盘+达标率 · AI用量周/月 · 平均交付天数代理口径 · 近7天活跃 · 卡住信号 · 最近动态）；【N2】端点 query 数与成员数无关——用 GROUP BY 一次聚合，handler 内不循环 N 次调 query；make sqlc 无 drift（绝不手改 generated）；go test ./internal/handler/ 绿；端点配 zod + parseWithFallback + ≥1 喂坏数据测试。
  • 波次1·前端：新路由 /[workspaceSlug]/team；顶部「团队 | 项目」tab；成员卡网格 web + desktop 双端渲染；团队页与卡落 packages/views/team/，query 落 packages/core/team/（core 零 react-dom/next，views 零 next）；【N4】卡片维度为封闭清单（= 通俗命名那几项，钉死数量），hover tooltip 至少落 1 条 jsdom 断言。
  • 波次1·默认落地改 /team（B1·全 callsite 扫荡，单点真相）：改集中 home 概念 packages/core/paths/paths.ts root() + 全部受影响 callsite 一致落 /team（paths/resolve.ts、app-sidebar 切 workspace、modals/create-workspace、apps/web onboarding、workspaces/new、issues/page、apps/desktop window-overlay ×2、desktop routes.tsx index Navigate）；验收：无任何入口仍默认落 /projects（grep 残留为 0）。
  • 波次2·侧边栏（B3·复用既有机制，不造第二真相源）：搜索 + 新建issue 移到 SidebarFooter 仍可正常触发；收缩复用既有 shadcn SidebarProvider 折叠态（collapsible='icon' 图标轨 + cookie sidebar_state 持久化），dashboard-layout.tsx 透传 defaultOpen；不新建 Zustand sidebar store；持久化跨重启生效。
  • 波次3·主从布局（B2·可选回调不破坏共享面）：board-card / list-row 加可选 onSelect/onIssueClick 回调、默认仍 fallback 到 AppLink；仅 project-detail 传 select 回调，点 issue → 保留「侧边栏 + 项目栏 + issue详情」（受控 selectedIssueId + URL 同步，复用 IssueDetail 的 onDelete/onDone）；my-issues / issues-page / actor-issues-panel 三个共享面导航不受影响；board 与 list 两视图都生效；深链 /issues/[id] 仍可全屏直达；desktop 经 URL param 同步并对齐 Route categories/Tab isolation 约定。
  • 全程：make check 全绿；原子 Conventional 提交（feat(team)/feat(sidebar)/feat(issues)）；每段 AI diff commit 前过人眼；严守包边界（页放 views 双端挂载不复制、UI 用 packages/ui shadcn/Base UI）。

方案

## 来源 + 评审 设计经 brainstorming 与用户对齐并批准（两版团队卡 mockup：高密度 + 通俗命名）。v1 经第二 session staff-eng 评审 → changes-requested（B1 默认落地 blast radius / B2 卡片点击波及 4 共享面 / B3 侧栏折叠与既有 shadcn 重复 + 落点纠错），本 v2 全部折入。 ## 仓库角色与数据地基 tc-multica 是平台本体（Go + pnpm/Turborepo monorepo）。关键洞察：团队卡的「平均交付天数 / 复盘达标率 / 本周完成 / AI 用量」恰是 cap-hardening P0 交付度量层的指标按【人】切——与 TEA-55 共享同一数据地基，是交付度量的「按人视图」。 ## 波次1 · 团队总览页（最重，含后端） 后端聚合 query（数据都在表里，只补只读聚合，镜像 task_usage.sql，绝不手改 generated；【N2】GROUP BY 一次聚合、handler 不循环 N 次）：项目 lead/DRI 数（project.lead_type/lead_id/dri_user_id）、参与项目数、issue 按 assignee+status 分组、agents by owner + 运行中（AgentTaskQueue active）、autopilot by created_by、squad 归属（squad_member）、项目进度、活动 feed（复用现有 activity handler）。成员列表（ListMembersWithUser）+ token 用量（ListDashboardUsageByPerson，本就挂 RequireWorkspaceMember、当前已全员可见）现成复用。平均交付天数走 issue.FirstExecutedAt→CompletedAt 代理口径并标注，精确口径依赖 cap-hardening W1.5 的 issue_event 表（确认不存在，本期用代理）。新 handler server/internal/handler/team.go，注册单聚合端点 GET /api/team/overview（X-Workspace-ID header · RequireWorkspaceMember）。前端 packages/views/team/（team-overview-page + member-card）+ packages/core/team/queries.ts；web + desktop 双端路由。 ## 波次1 · 默认落地改 /team（B1·全 callsite） 现状：workspace home 集中编码在 packages/core/paths/paths.ts root()（'${ws}/projects'，注释「Workspace home = unified project tab」），并被 9+ callsite 硬引用。改默认落地 = 改 root() 这一单点真相 + 扫所有 callsite（paths/resolve.ts 登录落最近 ws、app-sidebar 切 workspace、modals/create-workspace、apps/web onboarding、workspaces/new、issues/page、apps/desktop window-overlay ×2、desktop routes.tsx index 的 <Navigate to="projects">）使其一致落 /team。这是本波次最易超 appetite 处，单列为工作项、grep 验残留为 0。 ## 波次2 · 侧边栏（B3·复用 shadcn，不造 store） 改 packages/views/layout/app-sidebar.tsx：搜索（SearchTrigger）+ 新建issue 从 SidebarHeader 移到 SidebarFooter。收缩：直接复用 packages/ui/components/ui/sidebar.tsx 的 SidebarProvider 既有折叠态（open/setOpen/toggleSidebar、collapsible='icon' 图标轨、折叠态已写 cookie sidebar_state 持久化、宽度 localStorage），在 dashboard-layout.tsx:35 透传 defaultOpen（现未透传）。不新建 Zustand sidebar store（违反「prefer existing patterns / no duplicate state」）；若后续确需跨设备同步再单独立项。 ## 波次3 · 项目主从布局（B2·可选回调 + 落点纠错） 现状：project-detail.tsx 已是 3 面板 ResizablePanelGroup（list rail｜content｜properties），点 issue 经其 import 的 BoardView/ListView（项目-detail.tsx:44-45）→ 卡片组件 board-card.tsx / list-row.tsx（实际在 packages/views/ISSUES/components/，非 projects/components）的 AppLink 整页跳 p.issueDetail()。这两个 View 被 4 处复用（project-detail / my-issues / actor-issues-panel / issues-page）。改：给 board-card/list-row 加可选 onSelect?/onIssueClick? 回调、默认仍 fallback AppLink（不破坏另外 3 个无主从面板的面）；仅 project-detail 传 select 回调 → 更新 selectedIssueId（store + URL 同步）→ 中间 content 面板从 board/list 切成内嵌 IssueDetail（issue-detail.tsx:657 暴露 issueId/onDelete/onDone，可受控复用，onDelete/onDone 清选中）。desktop routes.tsx 已有 issues/:id 全屏深链（保留供外部直达）；主从的 URL 同步用 ?issue= 挂 projects/:id，并对齐 CLAUDE.md Desktop Route categories / Tab isolation。 ## 三个技术决策（已定） ① 团队页独立路由 /team；② 数据走单个聚合端点 /api/team/overview 一次拉全员；③ 主从用受控 selectedIssueId + URL 同步（兼顾深链与 desktop tab 隔离）。 ## 已决策记录 - 隐私（N1）：团队页「全员可见全员」per 用户明确选择；与现状 /api/dashboard/usage/by-person 已全员可见口径一致，非回归。社会成本（卡住/交付速度按人公示）已知悉，用户已拍全员可见。 - 默认落地（B1）：per 用户明确要求「默认进入显示团队页」，保留 /team 默认，按全 callsite 正确实施（非撤回）。 ## 硬约束 包边界（core 零 react-dom/next、views 零 next、整页放 views 双端挂载不复制、UI 用 packages/ui Base UI/shadcn）· TDD（测试放对包：core→Node、views→jsdom、web→框架 mock、Go→go test）· drift-defense（新端点 zod + parseWithFallback + 喂坏数据测试，不裸 as）· 多租户（query 过 workspace_id、UUID 走 parseUUIDOrBadRequest、X-Workspace-ID header）· sqlc 工作流（改 queries/ 跑 make sqlc，不手改 generated）· 原子 Conventional 提交、每段 diff 过人眼、每波次以 case 收尾。 ##
