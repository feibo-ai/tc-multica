目标

Goal
为 Multica 新增一个『个人 × runtime』粒度的 token 用量维度，覆盖本机【全部】Claude Code 对话 —— 包括从未经 agent_task_queue 派发的 ad-hoc 本地会话（今天对所有报表 0 行不可见，见 handler/runtime.go:84-87 自陈盲区）。做法：daemon 就地解析本地 transcript 日志（【只读数字、绝不读 message.content】），按 (message.id, requestId) 去重并经 server 端排除已执行任务的 session 去双算，归到 runtime.owner_id（= 注册该 daemon 的用户），经新 runtime 级端点上报到新的 task-less 用量表，再汇入复用控制骨架的并行 rollup，最终在 dashboard 露出一个【包含这部分此前不可见本地用量】的 per-person 总数。v1 范围：JSONL 解析、可插拔 Collector 框架（Claude 为首个 adapter）、只向前、daemon-owner 归属（含 owner_id 可靠性修复）。

完成标准

  • 【本地会话可见】在 Multica 之外跑一个 ad-hoc claude 会话（无任务），其 token 在一个 rollup tick 内出现在用量里，归到 daemon owner 的 person × runtime（= 注册该 daemon 的用户；今天的 mul_ PAT 路径成立）—— 跑一个一次性 claude 会话即可演示其 token 落账。
  • 【不双算】Multica 派发的任务其 transcript 不被重复计数：server 端 rollup 经 LEFT JOIN agent_task_queue 排除已执行任务的 session_id（session_id 已进写链）—— 对比一次 mounted run 前后的总数与它已产生的 task_usage 行可证（允许 in-flight 窗口内短暂双算、完成后下一 tick 自愈）。
  • 【用量正确性·评审 B1 修正】对含同-(message.id, requestId) 重复行的 fixture JSONL，去重后求和 = ccusage 参考值（不被重复行放大为 ~3.46×）；落盘 JSONL 无『流式 0 占位』病（那是 stream-json wire 格式的病）。
  • 【隐私】上报载荷与落库行只含 {session_id, timestamp, model, provider, 四个 token 数}；测试断言 upload 结构体永不读入 message.content / prompt 文本。边界 doctrine 记入 decisions/。
  • 【幂等 + fail-soft】对同一批文件重跑 scanner 0 新增行（复合唯一键生效）；一条畸形 JSONL 行被跳过而非致命（daemon 继续 tick）。
  • 【可插拔】Collector 接口有 ≥1 个具体 adapter（Claude）且契约文档化，加 Codex = 新写一个实现 Scan 的文件、不动管线。
  • 【端点受护】POST /api/daemon/runtimes/{id}/usage 经 requireDaemonRuntimeAccess 解析 runtime、过滤 workspace_id、校验 body；UI 读端点经 zod + parseWithFallback，并有一个 malformed-response 测试。
  • 【per-person 读出】dashboard 显示一个 per-person（× runtime）token 总数，同时含 mounted 任务用量与未挂载本地会话；字段缺失时优雅降级。
  • 【归属修复·评审 B3】owner_id IS NULL 的 ambient 行落『未归属』桶而非静默归 owner；032 回填为 workspace-owner 的旧 runtime 在真实用户 daemon 重注册后归属自愈（现有 COALESCE upsert 已提供）。
  • 【绿管线】make check 通过（typecheck / TS test / Go test / E2E）。
