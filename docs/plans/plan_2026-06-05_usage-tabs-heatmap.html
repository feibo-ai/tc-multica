目标

Goal
为 Usage（用量）页把下方「Cost by agent」排行榜区域改造成 👤 用户 | 🤖 智能体 二分 tab，并为两侧引入 GitHub 贡献图式热力图——全部基于 v1 已落的现有 hourly rollup（task_usage_hourly / ambient_usage_hourly），零 daemon / collector / migration / 新表 改动。用户 tab = 仅 ambient（本地 CLI）用量，按人按费用排行 + 点开看该人 26 周热力图（费用/Token 可切）；智能体 tab = 复用现有 by-agent 榜 + 点开看该 agent 26 周 task 热力图。新增 3 个只读端点（都在现有表上 GROUP BY），沿用 zod + parseWithFallback + malformed 测试、workspace_id 多租户过滤、owner NULL → 未归属 语义。落地 design 文档的 Phase 1（D1–D6 已锁，不重开）。

2. 范围后果：PersonUsageList 的去留（D2 直接推论，提请评审确认）

D2 锁定「干净二分：用户 tab = ambient 仅此」。现有 PersonUsageList 展示的是 task+ambient 合并的人总账——既非用户 tab（ambient-only）也非智能体 tab，与二分心智模型冲突。故 页面上移除 PersonUsageList 的渲染，其「人」维度由新的 ambient-only 用户 tab 取代。
API 边界保留：GET /api/dashboard/usage/by-person 端点与 dashboardUsageByPersonOptions hook 保留不删——老版本 desktop 安装仍渲染旧 PersonUsageList、会调它（API Response Compatibility：装机端比后端旧）。只在当前 views 里停止渲染该组件，不动端点。这是有意的「替换 UI、保边界」而非删 API。

3. 后端：3 个新只读 query + 3 个 handler（Go）

全部加在 queries/task_usage.sql（与现有 dashboard 读端并排，ListDashboardUsageByPerson 就在那读 ambient），make sqlc 重生，handler 进 dashboard.go，路由进 server/cmd/server/router.go。
用途新 query形状 / 关键 SQL路由
用户榜（人均 ambient，保留 model 以便客户端按模型算费用）NEWListDashboardAmbientUsageByPersonSELECT rt.owner_id, model, SUM(四 token) FROM ambient_usage_hourly auh LEFT JOIN agent_runtime rt ON rt.id=auh.runtime_id WHERE auh.workspace_id=$1 AND auh.bucket_hour>=@since GROUP BY rt.owner_id, model。无 tz 切天（tz 仅用于 Go 算 since）。NULL owner 落「未归属」桶。注：现有 by-person 塌了 model 只能算 token、算不了 cost，故必须新写保 model 的版本。GET /api/dashboard/usage/ambient/by-person?days=&tz=
用户热力图（某 owner 的 ambient/天）NEWListDashboardAmbientUsageDailySELECT DATE(bucket_hour AT TIME ZONE @tz) AS date, model, SUM(四 token) FROM ambient_usage_hourly auh LEFT JOIN agent_runtime rt ON rt.id=auh.runtime_id WHERE auh.workspace_id=$1 AND auh.bucket_hour>=@since AND ((sqlc.narg('owner_id')::uuid IS NOT NULL AND rt.owner_id=sqlc.narg('owner_id')) OR (sqlc.narg('owner_id') IS NULL AND rt.owner_id IS NULL)) GROUP BY DATE(...), model。narg owner_id 为 NULL ⇒ 未归属桶（该端点恒按单一 owner 或未归属过滤）。GET /api/dashboard/usage/ambient/daily?owner_id=&days=&tz=
智能体热力图（某 agent 的 task/天）NEWListDashboardAgentUsageDailySELECT DATE(bucket_hour AT TIME ZONE @tz) AS date, model, SUM(四 token) FROM task_usage_hourly WHERE workspace_id=$1 AND agent_id=@agent_id AND bucket_hour>=@since GROUP BY DATE(...), model。索引 idx_task_usage_hourly_workspace_agent_time 覆盖。GET /api/dashboard/usage/by-agent/daily?agent_id=&days=&tz=
智能体榜（agent 总额）已有ListDashboardUsageByAgent不动，直接喂智能体 tab 左榜。GET /api/dashboard/usage/by-agent
Handler 约定（硬约束）：
  • 三个 handler 都 resolveWorkspaceID(r) → WHERE workspace_id=$1（多租户）；resolveViewingTZ(r) + parseSinceParamInTZ(r, default, tz)（沿用现有 days/tz 截断）。
  • UUID 边界（CLAUDE.md #1661 约定；此为纯读 query、无 Delete*/Update*，#1661 的零-UUID 隐患本不适用，但实现细节须钉死）：agent_id 恒 UUID → parseUUIDOrBadRequest(w,s,"agent_id")，非法即 400 返回。owner_id 特例：缺省/"" ⇒ 直接构造 pgtype.UUID{Valid:false}（SQL NULL）传 narg，绝不把 "" 过任何 parse 函数（util.ParseUUID("") 会报错/产无效值），不当 400；非空才 parseUUIDOrBadRequest。不变式：该 daily 端点无「全 owner」模式（WHERE 只有 单 owner 或 未归属 两支），代码注释写明，否则 "" 会在「未归属桶」与「不过滤」间二义。注意 r.URL.Query().Get("owner_id") 对 ?owner_id= 与整个 key 缺失都返回 ""，二者皆走未归属。
  • 两个热力图 handler 的响应同形（{date, model, 四 token}），共用一个 response struct。NULL owner 经 row.OwnerID.Valid 判定 → 上线 ""（沿用 dashboard.go 现有 236–242 写法）。
  • Go handler 测试（server/internal/handler/，自建 fixture）：① workspace 过滤（别的 ws 不串）；② tz 切天正确；③ owner_id="" → 未归属行；③b 整个 owner_id key 缺失（非空串）→ 同样落未归属；④ 非法 agent_id → 400。
