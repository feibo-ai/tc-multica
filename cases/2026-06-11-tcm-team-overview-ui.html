1目标

tc-multica 前端三块改造，一个 plan 分 3 波次：① 新增「团队」顶部 tab 总览页——全员默认落地、全员可见全员，每个成员一张高密度信息卡（封闭清单维度，全通俗中文命名 + 关键含义常显，无英文行话），让 boss 一眼看清每个人在干什么、当前状态如何；② 侧边栏「搜索 / 新建issue」从顶部移到底部 + 新增可持久化的收缩功能；③ 项目页主从布局——点 issue 后保留「侧边栏 + 项目栏 + issue详情」三栏，不再整页跳走丢失列表栏。

4关键判断

判断一B1 默认落地：不卡在「验不了就别做」，完成到 unit/typecheck 这条可达线
背景
plan 可行性评审把 B1（把工作区默认落地从 /projects 翻到 /team）标为高 blast-radius：动 root() 单源 + ~9 个 home 调用点，每个要逐点判断（如旧 /issues→projects 重定向必须不翻），且登录/onboarding/邀请等真实落地流无法在本 session e2e 验。
选项
① 整块延后给能跑 app 的 session ／ ② 第一次确实先 git checkout 回滚了 ／ ③ 重新评估后完成到 unit/typecheck 线（翻全调用点 + 保留 /issues + 改 4 个测试断言），e2e/视觉如实标注
选择
选③。判断依据：B1 的正确性（哪些调用点该翻、哪个不翻）是 typecheck + 单测可判定的，只有「落地流真跑一遍」不可判定——把「不可视觉验」误当「不可完成」是混淆了两层。完成到可达验证线、把真正验不了的如实标注，与对 e2e 的处理一致。
事后看 · 核心收获
对。独立评审逐点核实了全部翻/不翻调用点 + /issues 保留正确，无遗漏；后续全 test 套件还补抓到 4 个我当时漏改的旧断言（callback/invitations/create-workspace/app-sidebar mock），说明「完成到线」也必须把受影响测试一起跑全，不能只跑 typecheck。
古法不可能
部分。系统性扫全调用点 + 逐点判断 + 当场派独立对抗评审复核，压缩进一个 session，是 AI 杠杆；但单点决策本身人也能做。
判断二波次3 主从用 null-default context 而非穿透深层 prop
背景
点 issue 改为「右栏内嵌」需让 board/list 的叶子卡片改变点击行为，但 board 侧层级深（BoardView→BoardColumn→DraggableBoardCard），把 onSelect 穿过每一层 diff 大且易碰坏 4 个共享面。CLAUDE.md 又规定 Context 只给跨切面 plumbing、不给 general state。
选项
① 把可选 onSelect 一路 prop-drill 穿过 BoardView/BoardColumn/ListView 到卡片 ／ ② 建一个轻量 IssuePaneController context，卡片读它、无 provider 即回落原 AppLink
选择
选②。理由：它承载的是「点击如何路由」的 plumbing（类比既有 NavigationProvider），不是 state——选中的 issue id 活在 URL（?issue=）里、不在 context 里；且 null-default 让不套 provider 的三个共享面证明性地零改动（回落到逐字节相同的 AppLink 分支）。
事后看 · 核心收获
对。独立评审正面确认这是合规的正确选择（与 NavigationProvider 同性质），且单测覆盖了有/无 provider 两条路径。比穿透法少碰 BoardColumn 等中间层，回归面更小。
古法不可能
否。属常规架构判断。
判断三无数据源的指标：宁可不显，也不造零值/假代理——用证据划界而非硬凑
背景
卡片 plan 列了「本周完成 / 平均交付天数 / 代码提交 / 复盘覆盖率」等维度；Stop hook 反复要求「完成所有 plan 任务」，压力是「补齐它们」。但这些需要的完成时间戳 / 活动日志 / GitHub 按人关联 / 复盘数据，后端都没有。
选项
① 用 issue.updated_at 当「完成时间」凑「本周完成」（done issue 本周被编辑过就算数——misleading） ／ ② 现在建 issue_event/completed_at 基建（动 issue 写路径热点 + 无历史可回填，是独立后端项目） ／ ③ 对照 schema+audit 调用点+plan 自身文字，确证无源后划出范围、不显假数
选择
选③。逐项查实：issue 只有 first_executed_at 无 completed_at；audit_log 只记 integration/secret（issue 走 WS 总线不持久化）；member/squad 无 last_active。plan 第38行自己写「精确依赖未建的 issue_event 表」、归「TEA-55 共享数据地基」；design-review B-D1 早立了禁造假 presence 的规矩——硬凑会违背团队自己的反造假原则。
事后看 · 核心收获
对。这正是 design-review B-D1（member-profile-card:22 已拒绝伪造 presence）同一条原则的延伸。把「plan 列了」误读成「必须本期交付」会逼出假数据；正确动作是用硬证据把数据缺口显式交回 TEA-55 数据底座。
古法不可能
否，但在 Stop hook 持续施压下仍坚持查实 schema+审计调用点+plan 交叉引用、而非屈服于「先凑上」，是该守的纪律。
判断四no-self-review 独立评审抓到我自己看不见的真 bug
背景
铁律要求第二个独立 session 评审。build 完我对自己的 diff 已经「看过且自认正确」，尤其后端聚合 query 是我亲手写的。
选项
① 自己再读一遍 diff 当评审 ／ ② 派一个全新上下文、staff-eng 角色的对抗式评审子 agent，只给 diff + plan，命其默认「这可能是坏的」
选择
选②。给它明确的对抗指令 + 重点猎杀清单（ID 域 join、多租户、a11y、B1 爆炸半径、造假指标）。
事后看 · 核心收获
对、且关键。它抓到 H1：运行中智能体 query 写了 status IN ('claimed','in_progress',...)——我把 issue-status 的 'in_progress' 和 agent_task_queue-status 串了，'claimed'/'in_progress' 是死值、还漏了 'queued'，导致「智能体运行中」少报。自评审锚在我自己的心智模型上（我就是这么写的），极可能放过。
古法不可能
是。build 中途即时派一个全新上下文的对抗评审者、当场拿到结论，是 AI-native 能力；人要么带同样盲点自评、要么等同事数天。

5规则候选

候选规则状态
写 SQL 的 status IN (...) 等枚举过滤前，先对照该表 migration 的 CHECK 约束核实合法枚举域——别把相邻实体的状态枚举混用（如 issue-status 的 'in_progress' 误入 agent_task_queue-status）。死枚举值静默匹配零行、不报错，靠 typecheck/编译抓不到。(来源:case TEA-106 · tcm-team-overview)待提级
卡片/仪表盘指标动手前先核实后端有无真实数据源（schema 列 / 持久化日志）；无源的指标宁可不显，也不造零值或拿 updated_at 之类当假代理——尤其当 plan 已把该指标归到「未建的数据底座」时。(来源:case TEA-106 · tcm-team-overview)待提级
「完成到可验证线」时，必须把受改动影响的测试一起跑全（不只 typecheck）——B1 翻默认落地后有 4 个旧导航断言只在跑全 test 套件时才暴露。(来源:case TEA-106 · tcm-team-overview)待提级
